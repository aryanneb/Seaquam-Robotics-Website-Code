Coding your drive


Drive is a code for your robot to move and controlled by your VEX V5 Controller.
So in other words, you need to program your robot like a RC car so that the driver in your team can drive.

-So what should I code?
There are several different types of drive that you can code, and you need to ask your driver for their preference first.

1.Tank drive
Tank drive is a drive where your left and right joystick on the controller moves each side of the robot.
For example, when you just tilt your right joystick, it will only move your right wheels/motors on the robot.

2.Arcade Drive
Arcade drive is a drive where you have one joystick to do everything, from turning, curve and going straight.

These are the two major types of drive if you use a normal chassis design, and a normal omnidirectional wheels on your robot.

-How do I code this?
There are example codes officially from VEX Robotics, you can find a "open examples" button from the "Files" tab on your VEXcode/VEXcode PRO IDE.
However, if you are missing some information, go to the Useful Templates page on each of the coding IDE you use.

VEXcode Pro

How to Code


I will go over some basic steps for the new coders to VEXcode PRO.

1. Create your project

What you first do when you code, you go to the File > New tab on the top-left navigation.
Name your project, and save to where you can easily access from your desktop.
I usually save it in a file folder directly created in the desktop.

For competition programs, make sure you create a new project by selecting
File > Open Examples... > Competition Template.
This way, you don't need to worry about going through code inspections, unless you do something really wrong.


2. Fonfigure your VEX devices

After you have created your program, now you want to configure your VEX devices, such as motors, controllers, and sensors if you use them.

You click the icon which looks like a VEX Brain Port, located on the top-right.
Then, you click "add a device", and select the device you are going to use.
For controllers, you just click "Controller", and then "done" after.
For motors, you first select which VEX Brain Port you are using, then select "Normal" or "Reversed" corresponding to the direction the motor is facing.
Each sensors have a different set-up steps, but it is not hard.

3. Code your basic functions

So, you are now finally ready to code.
First, it is strongly recommended to create any repetitive patterns of code made into functions.

For examples, you can go check the page.

4. Code your drive/autonomous

After you created your functions, now you use them to code your drive and autonomous!
There are various things you can try out, but first, go check the page to learn how coders in Seaquam Robotics code their programs.

Useful Templates


This page will give examples on basic, useful functions that you can implement into your code!

Set Velocity All function:
void setVeloAll(int leftMotorVelocity, int rightMotorVelocity) {
LMtr.setVelocity(leftMotorVelocity,percent);
LMtr2.setVelocity(leftMotorVelocity,percent);
RMtr.setVelocity(rightMotorVelocity,percent);
RMtr.setVelocity(rightMotorVelocity,percent);
}
This code sets velocity (speed) of the all four drive motors.
There are two parameters for the velocity; One for the left motors and one for othe right motors.
This way, you can use this for turning and spinning as well, aside from going back and forth.
--------------
void stopAll(void) {
LMtr.stop();
LMtr2.stop();
RMtr.stop();
RMtr2.stop();
}
This code stops all of the drive motors. You will use this a lot, especially for the autonomous codes.
--------------
void driveWithDeg(int leftMotorVelocity,int leftDeg,int rightMotorVelocity, int rightDeg) {
setVeloAll(leftMotorVelocity, rightMotorVelocity);
LMtr.startRotateFor(leftDeg, degrees);
LMtr2.startRotateFor(leftDeg, degrees);
RMtr.startRotateFor(rightDeg, degrees);
RMtr2.startRotateFor(rightDeg, degrees);
}
This code uses setVeloAll(int, int); to set velocity for all motors, and then moves for the assigned degrees value.
This is also used a lot in the autonomous codes.

Troubleshooting


This section answers some frequently occuring coding problems.


- My code shoots out errors!
Read the errors carefully. You might be missing a brace, bracket or a semicolon.
You might have wrong data type values in your functions.

- My robot does not go straight!
Are you using a time-based code? That makes the code inconsistent, making the robot move inaccurately.
Are your robot built straight? Hardware issues can affect your robot.

Design Process


This goes through what is a Design Process for coding and its purpose.

So, what is it?
Design process is a series of steps which you go through when you are trying to design and engineer anything.
This applies to coding too. Your speed and quality of coding will bump up when you have this.

There are different models from various places and I will encourage you to dive in and search for the one that best fits you.

Generally and usually, the design process has sections for planning, testing, trying and improving the item, and each of these are divided smaller in different models.
You will go through these process to ensure you are productive and it tells you what to do like a manual.

Establishing a good desing process means you are having a way better productivity and performance as a coder.
So find your way of coding, check out some different styles created by big corporations and some awesome engineers.

Design

Your Way Of Coding


This section will provide some example of a coding process, followed by the actual Seaquam Robotics Coder.

Process of Ryu Sado - 9181C

I usually tend to follow some very simple and short design process that is easy to follow and very fast to move on.

There are four steps:
1.Plan&Analyze - You make your general plan as you collect information on what would be effective and efficient.
2.Test&Teardown - You tear down your general plan into very small sections, like a single function, and then test them to perfection.
3.Trying&Refining - You merge all the small sections you created at step 2, and then you will go through the whole thing until you don't find any flaws.
4.Cut-off&Improve - Your step 3 continues until your due date (tournament day) comes. After the tournament, you will take the result and give ideas and plans for better improvement.

This design process was very successful for me, since I had established this process in my first year of VEX EDR robotics, and I managed to go to the worlds championship through the skills challenge.

I hope this fits you too!

Documentation


This page explains what documentations are for.

There are several reasons:
1. You find your past mistake.
Human memories are not that reliable - therefore, it is better to note what you tried, and failed so that when you try it again,
you do not need to repeat the same mistake.

2. Good for presentation.
It is way better to have a big log of your work to show to the judges, rather than just showing a line of code in your laptop.
It gives them the better idea of what you have worked on, and your design process.


3. You find your ancient code
You can quickly find how to code your awesome code that worked 3 tournaments ago. You might be able to re-use and re-cycle your brain power from the past!
